Create and maintain a structured task list with subtask support for the current session. **ALWAYS use this tool at the start of every task** to break the work down into steps. This is mandatory — every request gets a todo list.

## Task fields
- `content` (required): Brief task description
- `description` (optional): Detailed notes, context, or acceptance criteria
- `status`: pending | in_progress | completed | cancelled
- `priority`: high | medium | low
- `parentId` (optional): Set to another task's `id` to make it a subtask. Subtasks nest under their parent in the display.
- `tags` (optional): Labels like bug, feature, refactor, docs, test
- `dependsOn` (optional): Task IDs that must be completed before this one

## Mandatory workflow
1. **On every request** — Create a todo list breaking the work into steps. Always do this first.
2. **Mark `in_progress`** — Before starting any step, mark it in_progress (exactly one at a time)
3. **Mark `completed`** — After finishing a step, mark it completed with verification
4. **Mark `cancelled`** — If a step becomes unnecessary, mark it cancelled
5. **Subtask complex work** — Any task with multiple sub-steps gets subtasks via parentId
6. **Reflect after completion** — When all tasks are done, call `reflect` to capture what was learned

## Examples
- User says "fix the login bug" → todowrite: ["reproduce the bug", "find root cause", "implement fix", "write test", "verify"]
- User says "add dark mode" → todowrite: ["design color scheme", "implement CSS variables", "add toggle switch", "test light/dark", "run lint"]
- User says "what's the weather?" → todowrite: ["check weather for location", "report back"] (always, even for simple tasks)

## Rules
- Break EVERY request into at least 2 steps. No exceptions.
- Keep exactly one `in_progress` while work remains
- Add tags and dependsOn when they help with organization
- If a step reveals more work, add new steps to the list
- When all steps are done, call `reflect` to create a skill from the experience
